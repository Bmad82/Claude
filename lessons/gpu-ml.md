# GPU / ML-Modelle

- **Device-Detection mit VRAM-Fallback:** `torch.cuda.mem_get_info()` gibt `(free, total)` in Bytes. Pattern: `auto`-Mode prüft `free_gb >= threshold` (z.B. 2 GB für MiniLM + Cross-Encoder); `cuda` erzwingt GPU mit Fallback auf CPU bei ImportError/kein CUDA; `cpu` überspringt alles. In einer eigenen `get_device()`-Funktion isolieren, `_cuda_state()` als mockbares Hilfs-API definieren — sonst sind Tests entweder GPU-abhängig oder brauchen `torch` als Test-Dependency (Zerberus P111)
- **torch + CUDA nur mit Extra-Index:** `pip install torch` zieht Default die `+cpu`-Variante. Für GPU: `pip install torch --index-url https://download.pytorch.org/whl/cu121` (oder passende CUDA-Version). Nach `pip list | grep torch` prüfen — `2.x.y+cpu` zeigt die CPU-only Variante. Ohne CUDA-torch bleibt `torch.cuda.is_available()` IMMER False, auch bei installierten NVIDIA-Treibern (Zerberus P111)
- **SentenceTransformer / CrossEncoder nehmen `device=` im Konstruktor:** `SentenceTransformer("model", device="cuda")` bzw. `CrossEncoder("model", max_length=512, device="cuda")`. Ab `sentence-transformers >= 2.2`. Keine nachträgliche `.to(device)`-Verschiebung nötig. Mit `inspect.signature(CrossEncoder.__init__)` verifizieren, bevor man sich auf den Parameter verlässt (Zerberus P111)
- **Cross-Encoder auf CPU bei 30+ Chunks = 30-50 s.** Bei SSE-Streaming auf Mobile = Timeout + Retry-Sturm. GPU bringt 5-10× Speedup. Bei Latenz-Problemen IMMER zuerst prüfen ob das Modell auf CPU statt GPU läuft — Logs zeigen oft `No device provided, using cpu` (Zerberus P111)
- **VRAM-Budget planen:** Whisper ~4 GB, BERT-Sentiment ~0.5 GB, MiniLM ~0.5 GB, bge-reranker-v2-m3 ~1 GB. Bei 12 GB Karten passt alles gleichzeitig; bei 8 GB Karten muss man zwischen Embedding + Reranker priorisieren oder quantisieren
- **CUDA-Driver ist rückwärtskompatibel:** `nvidia-smi` → `CUDA Version: 13.1` bedeutet nur, dass der Treiber Applikationen bis zu CUDA 13.1 unterstützt. Die tatsächlich installierte CUDA-Toolkit-Version ist separat. Man kann problemlos torch mit `cu124` auf einem 13.1-Treiber installieren — der Driver akzeptiert alle älteren CUDA-Runtime-Versionen (Zerberus P111b)
- **typing-extensions-Ripple nach torch-Reinstall:** `torch 2.5.1+cu124 --force-reinstall` installiert `typing_extensions 4.9.0` als Dependency (genau die benötigte), aber andere Pakete (z.B. `cryptography>=46`) verlangen >=4.13.2. Nach jeder torch-CUDA-Install sofort `pip install --upgrade "typing-extensions>=4.13.2"`. Symptom beim Auslassen: `ImportError: cannot import name 'TypeIs' from 'typing_extensions'` (Zerberus P111b)
- **`+cuXXX`-Suffix in `pip list` ist der Proof:** `torch 2.5.1+cu124` = GPU-Wheel. `torch 2.5.1` (nackt) oder `torch 2.5.1+cpu` = CPU-Only. Während eines laufenden `pip install --force-reinstall` kann `pip show torch` temporär „Package not found" mit Hinweis auf `-orch`-Dir zeigen (Windows-File-Lock-Umbenennung) — erst nach Completion ist der neue Zustand verlässlich (Zerberus P111b)
